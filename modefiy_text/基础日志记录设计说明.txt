MESClient 基础日志记录设计说明

一、设计目标

本项目是生产执行系统客户端，日志不只用于程序调试，还要支持生产追溯。
基础日志需要回答四个问题：

1. 谁操作的：员工工号、员工姓名、角色、班次、工位。
2. 什么时候操作的：本机时间、登录会话。
3. 操作了什么：扫描、过站、PASS、NG、维修判定、导出、登录等。
4. 结果是什么：成功、失败、失败原因、后端返回信息。

注意：
1. 不记录密码。
2. 不完整记录 token。
3. 不在表格 data() 等高频函数里打印日志。
4. 业务日志优先记录关键状态变化，不记录普通 UI 绘制过程。


二、建议新增的基础日志模块

建议新增文件：

1. commFunc/logmanager.h
2. commFunc/logmanager.cpp
3. commFunc/businesslogger.h
4. commFunc/businesslogger.cpp

LogManager 职责：

1. 在 main.cpp 中初始化。
2. 通过 qInstallMessageHandler() 接管 qDebug/qInfo/qWarning/qCritical。
3. 写入普通运行日志文件。
4. 支持日志按日期分文件。

BusinessLogger 职责：

1. 专门记录生产业务日志。
2. 自动从 UserSession 里补充员工信息。
3. 自动补充时间、角色、工位、班次、sessionId。
4. 写入独立业务日志文件。

推荐日志目录：

logs/app_yyyyMMdd.log
logs/business_yyyyMMdd.log

app 日志示例：

[2026-05-31 22:40:01.123] [INFO] [main] app started version=...
[2026-05-31 22:40:03.456] [WARN] [HttpClient] network error requestId=...

business 日志示例：

[2026-05-31 22:41:10.321] action=PRODUCT_PASS employeeId=E10001 employeeName=张三 role=operator workstation=ST01 productSn=SN001 taskNo=MO001 result=PASS


三、员工信息设计

建议 UserSession 最少支持以下字段：

employeeId      员工工号
employeeName    员工姓名
userName        登录账号
role            角色
workstation     工位
shift           班次
sessionId       本次登录会话 ID

当前项目 UserSession 已有：

network/usersession.h
network/usersession.cpp

已有字段：

token
userName
role
userInfo

建议后续先从 userInfo 中兼容读取员工信息，例如：

employeeId   = userInfo.value("employeeId").toString()
employeeName = userInfo.value("employeeName").toString()
workstation  = userInfo.value("workstation").toString()
shift        = userInfo.value("shift").toString()

如果后端暂时没有返回这些字段，先保留为空，不影响日志框架。


四、日志级别建议

qDebug:
开发调试阶段使用，正式版本可以关闭或减少。

qInfo:
正常关键流程，例如程序启动、登录成功、请求发送、页面切换、业务操作成功。

qWarning:
可恢复异常，例如配置文件不存在、JSON 解析失败、网络超时、用户输入校验失败。

qCritical:
严重错误，例如日志目录无法创建、核心配置不可用、程序无法继续运行。

BusinessLogger:
生产追溯操作，不完全等同于 qInfo。凡是影响产品、任务、维修、质量结果的操作，都应该进入 BusinessLogger。

七、第一阶段最小实现范围

第一阶段建议只做这些：

1. LogManager 接管 Qt 日志并写入 logs/app_yyyyMMdd.log。
2. BusinessLogger 写入 logs/business_yyyyMMdd.log。
3. 登录成功/失败记录员工信息。
4. 产品 scan in、PASS、NG 记录业务日志。
5. 维修判定保存/提交记录业务日志。
6. 报表导出记录业务日志。
7. 网络请求记录 requestId、url、状态码、耗时。

这个范围已经能覆盖大多数生产追溯和问题排查。


八、第二阶段增强项

1. 日志文件按大小轮转，例如单文件超过 10MB 自动新建。
2. 只保留最近 30 天日志。
3. 增加日志导出按钮，方便现场把日志发给开发。
4. 业务日志使用 JSON Lines 格式，方便后续导入后台。
5. 重要业务日志同时上传后台，避免本机日志丢失。
6. 增加崩溃日志和未捕获异常日志。


九、软件运行调试日志设计

除了生产业务日志，还需要软件运行调试日志。
这类日志主要用于开发、测试、现场排查问题，重点回答：

1. 程序在哪个模块出错。
2. 哪个函数出错。
3. 出错时的关键参数是什么。
4. 错误是否影响用户继续操作。
5. 是否需要开发人员介入。

软件调试日志建议统一进入：

logs/app_yyyyMMdd.log

如果后续日志量变大，也可以拆成：

logs/app_yyyyMMdd.log
logs/network_yyyyMMdd.log
logs/ui_yyyyMMdd.log
logs/error_yyyyMMdd.log


十、软件日志建议字段

每条软件日志建议至少包含：

time            时间
level           DEBUG/INFO/WARN/ERROR/FATAL
module          模块，例如 Network、Config、UI、Model、File、PDF
file            源文件名
function        函数名
line            行号
message         日志内容
requestId       网络请求 ID，可选
errorCode       错误码，可选
errorText       错误说明，可选

推荐格式：

[2026-05-31 23:10:01.123] [WARN] [Network] httpclient.cpp:86 parseReply requestId=xxx json parse failed error=...
[2026-05-31 23:10:02.456] [ERROR] [Config] configmanager.cpp:30 loadConfig open config failed path=...

十二、软件日志级别使用建议

DEBUG：
临时开发调试，例如某个控件尺寸、某个分支是否进入。正式版本可以关闭。

INFO：
正常流程节点，例如启动、退出、页面打开、请求发送、请求成功。

WARN：
用户可以继续使用，但需要关注。例如配置文件不存在后自动创建、JSON 文件解析失败、图片不存在、校验失败。

ERROR：
功能失败。例如网络请求失败、导出失败、PDF 加载失败、页面打开失败。

FATAL：
程序无法继续运行。例如核心目录无法创建、核心配置不可用、关键依赖缺失。


十三、建议的日志宏

为了能自动带上 file/function/line，建议后续增加简单宏：

LOG_DEBUG(module)    qDebug().noquote()    << "[" << module << "]"
LOG_INFO(module)     qInfo().noquote()     << "[" << module << "]"
LOG_WARN(module)     qWarning().noquote()  << "[" << module << "]"
LOG_ERROR(module)    qCritical().noquote() << "[" << module << "]"

如果使用 qInstallMessageHandler，可以从 QMessageLogContext 中拿到：

context.file
context.function
context.line

这样每个日志点不用手动写文件名和行号。


十四、特别提醒

1. 生产追溯日志要稳定，不要随意修改字段名。
2. 普通调试日志可以调整，但业务日志字段要尽量保持兼容。
3. 员工信息要从 UserSession 统一读取，不要每个页面各自拼。
4. 网络日志不能记录 password、完整 token、身份证号等敏感信息。
5. 过站、PASS、NG、维修判定属于核心业务日志，优先级最高。
