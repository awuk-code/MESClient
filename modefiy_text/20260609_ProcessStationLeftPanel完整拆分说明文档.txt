2026-06-09 ProcessStationLeftPanel 完整拆分说明文档

一、拆分背景

原 ui/processstationleftpanel.cpp 同时承担了以下职责：

1. 左侧页面整体布局
2. 任务单信息、异常品信息、任务单状态三个信息区的 UI 创建与刷新
3. 扫码过站区域 UI 创建
4. 过站状态表数据维护
5. 产品扫码、PASS、NG、暂停、解除暂停业务流程
6. NG 异常登记、暂停原因登记、解除暂停原因登记弹窗 UI
7. 工时开始、暂停、恢复、结束计时
8. 任务状态实时统计刷新

这些职责混在同一个文件中会导致 ProcessStationLeftPanel 体积过大，后续继续改业务逻辑或 UI 样式时维护成本较高。
当前拆分遵循“小步拆分、行为不变”的原则：先拆 UI 与状态辅助类，再逐步考虑业务控制器。

拆分后：

- ProcessStationPassStatus 维护状态表和计时。

4. 过站弹窗

拆分前：

- NgAbnormalDialog、PauseReasonDialog、ResumeReasonDialog 定义在 processstationleftpanel.cpp 匿名 namespace 中。

拆分后：

- 三个弹窗类移动到 ProcessStationPassDialogs 模块。

5. 业务流程

拆分前：

- ProcessStationLeftPanel 处理所有业务流程。

拆分后：

- 业务流程仍保留在 ProcessStationLeftPanel。

- validateProductNotPaused
- productSnForPauseAction
- currentProductSnFromSelection
- updateTaskStatusRealtime

原因：

1. 这些方法涉及页面级业务编排，不只是单纯 UI。
2. 这些方法依赖外部信号 productSnScanned、productNgSubmitted。
3. 这些方法依赖 DisplayMode、任务信息、异常品信息、右侧 PASS 校验器。
4. 如果一次性拆出业务控制器，容易同时改动业务行为和 UI 结构，风险较高。

十二、当前拆分阶段结论

当前 ProcessStationLeftPanel 已从“全量 UI + 弹窗 + 状态 + 业务”收敛为“页面组合 + 业务编排”。

已拆出的职责：

- 信息区 UI：ProcessStationInfoSection
- 扫码过站 UI：ProcessStationPassWidget
- 状态表和计时：ProcessStationPassStatus
- 过站弹窗：ProcessStationPassDialogs

暂未拆出的职责：

- 过站业务控制
- 字段映射配置
- 页面级信号转发
- 显示模式切换

建议继续按“先控制器、再字段映射、最后模型化状态表”的顺序拆分。
